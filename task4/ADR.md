### <a name="_3bfxc9a45514"></a>**Функциональные требования**

1. Система должна предоставлять актуальные ставки депозитов для внутреннего колл-центра в режиме чтения.
2. Система должна автоматически формировать выгрузку ставок для партнёрского колл-центра.
3. Формат файла должен быть стандартным (например, CSV или Excel).
4. Должна быть возможность задавать расписание обновлений ставок (каждую ночь).
5. В выгрузке должны присутствовать: название продукта, срок, ставка, условия, а по возможности — персональные ставки по категориям клиентов.

### <a name="_qmphm5d6rvi3"></a>**Решение**
    Приведите диаграммы контекста и контейнеров в модели C4. Опишите там основные компоненты и интеграции всех элементов решения. 

Также опишите, какой логикой вы руководствовались в ходе принятия решений и выбора технологий. Не забывайте, что необходимо учесть все функциональные и нефункциональные требования.

Решения принимались исходя из требований бизнеса: обеспечить возможность консультации клиентов по актуальным ставкам через кол-центр и партнёрский кол-центр, минимизировать ручную работу и ошибки, использовать существующую инфраструктуру банка и безопасно передавать данные. Для внутренних сотрудников выбран прямой доступ к бэк-офису депозитов, для партнёров — выгрузка файлов (CSV/XLS) через защищённый канал.

### <a name="_bjrr7veeh80c"></a>**Альтернативы**
Опишите здесь наиболее важные альтернативные решения.
1. Возможность проработки общего API с партнёрским колл-центром, минуя необходимость передачи файлов, напрямую загружая в систему данные о ставках.
Не выбрано: интеграция через API потребовала бы больше времени на разработку и дополнительные затраты партнёра, которые банк должен был бы покрывать; партнёрский колл-центр на текущий момент поддерживает только работу с файлами.

**Недостатки, ограничения, риски**
Передача ставок партнёрам через файлы — возможны задержки или ошибки.
Человеческий фактор при обработке файлов.
Требуется контроль безопасности при передаче файлов.

Ограничения MVP:
Партнёрский колл-центр получает ставки только через файлы, прямого API нет.
Полная автоматизация процесса обновления ставок и интеграция с другими продуктами банка пока невозможны.

Риски безопасности:
Необходимо корректно шифровать все файлы и защищать каналы передачи.
Контроль доступа сотрудников к файлам с актуальными ставками.

Ограничения технологического стека:
Используются только технологии, по которым есть экспертиза.
